# i18n — next-intl (FR + EN)

L'app utilise **next-intl** (v4). Le site vitrine est traduit ; l'app connectée l'est aussi depuis le 11/07/2026 (migration terminée sur tout le périmètre identifié — voir `docs/changelog.md`). **Toute nouvelle chaîne visible par l'utilisateur passe par next-intl dès l'écriture**, dans `fr.json` **et** `en.json` — jamais de littéral FR dans le JSX, jamais de migration différée à un futur audit.

## Où vivent les traductions

- `messages/fr.json` (fait foi pour l'ensemble des clés) et `messages/en.json` — **toujours synchronisés** : toute clé ajoutée dans l'un doit l'être dans l'autre, même structure. Une clé manquante côté EN casse l'exécution next-intl.
- Config : `src/i18n/routing.ts` (locales + chemins localisés type `/tarifs`↔`/pricing`), `src/i18n/request.ts`, `src/proxy.ts`.
- `NextIntlClientProvider` monté une fois dans `src/app/[locale]/layout.tsx` → tous les composants client de l'app connectée appellent `useTranslations` directement, sans prop-drilling (vrai même pour les sous-composants au niveau module comme `WorkshopCard`/`ItemCard`, qui sont sous le provider).
- `src/i18n/types.ts` augmente next-intl avec `Messages = typeof fr.json` → `t('clé')` est vérifié par TypeScript (une clé inexistante casse `npm run build`, l'IDE autocomplète). Ne pas éditer ce fichier.

## Convention de nommage

Un **namespace par page/feature** (`profile`, `dashboard`, `workshop`, `settings`, `examen`, `garden`…), clés en **camelCase imbriqué** (`profile.subscription.manage`). Réutiliser les namespaces transverses existants (`nav`, `footer`…). Jamais de clé fourre-tout globale. Attention aux collisions de nom entre l'espace connecté et le marketing public — ex. `accountPricing` (page compte, tiers d'abonnement) est distinct de `pricing` (section marketing publique) ; `createWorkshop` (flux connecté) est distinct de `create` (section marketing publique).

## Les 3 patterns d'usage

1. **Composant client** (`'use client'`) **ou composant serveur non-async** : `const t = useTranslations('profile'); … {t('greeting', { name })}`.
2. **Composant serveur async / `generateMetadata` / server action** : `const t = await getTranslations('profile');` (`useTranslations` est interdit en async — règle des hooks).
3. **Interpolation** : `t('energy.sessions', { count: 8 })` avec la clé `"{count} sessions aujourd'hui"`. Pluriels/rich-text : syntaxe ICU de next-intl si besoin.

## Patterns établis (à réutiliser)

- **Fonction pure qui renvoie une chaîne** (ne peut pas appeler `useTranslations`, règle des hooks) : soit on passe la chaîne déjà traduite en paramètre, soit on déplace la fonction dans le composant pour fermer sur `t`.
- **Map de libellés indexée par un type** (ex. anciennes `RESPONSE_TYPE_LABELS`) : remplacer `MAP[type]` par une **clé dynamique type-safe** `t(\`responseType.${type}\`)` — valide car `type` est une union littérale (le template donne une union de clés existantes). Ne PAS faire `t('responseType.' + x)` (donne `string`, rejeté par le typage des clés).
- **Clé dynamique sur une valeur `string` non typée** (ex. une valeur de donnée stockée en base) : `t(\`status.${s}\`)` ne type-check pas si `s` est `string`. Utiliser un petit helper à `switch`/ternaire qui ne passe que des littéraux à `t`.
- **Collision de nom `t`** (le composant a déjà un paramètre/variable `t`, ex. `.map(t => …)`) : nommer le traducteur `tr`.
- **Mock data / valeurs stockées en base** (ex. données d'exemple destinées à venir du backend, ou une valeur `enum` stockée telle quelle type `exam.status`) : la donnée reste en FR, on traduit seulement à l'**affichage** via un helper dédié — ne pas i18n-iser la donnée elle-même.
- **Formats neutres** (placeholders type `#tag…`, `000000`) : pas besoin de traduction.

## Emails transactionnels (`src/lib/emails.ts`)

Un email se construit dans la langue du **destinataire**, jamais de celui qui déclenche l'action (l'émetteur d'une suppression peut naviguer en EN alors que le propriétaire notifié est FR ; un email lu plus tard n'a plus de contexte de requête). La langue de chaque compte est persistée dans Clerk `publicMetadata.locale`, synchronisée depuis l'URL par `DashboardHeader` (auto-sync au chargement/changement, source de vérité) et écrite via `setUserLocale` (`src/app/actions/profile.ts`).

- Chaque builder est **`async`** et prend `locale`, lue via `getTranslations({ locale, namespace: 'emails' })` — la locale explicite fonctionne hors contexte de requête (next-intl la passe en `localeOverride`, `src/i18n/request.ts` la résout). Textes dans le namespace `emails` de `fr.json`/`en.json`.
- **Un envoi par destinataire, chacun dans sa langue** : pour un email groupé (ex. tous les propriétaires), lire `publicMetadata.locale` par personne (helper `emailLocaleOf` dans `lib/workshops/lifecycle.ts`) et construire un email par destinataire dans un `Promise.all` — ne jamais construire un seul HTML partagé. Repli `?? 'fr'` (defaultLocale) si aucune préférence enregistrée.
- **Piège : pas de balise HTML dans un `t()` simple.** next-intl exige `t.rich` pour les balises, mais `t.rich` renvoie des nœuds React — inexploitable pour une chaîne HTML d'email. Donc les messages `emails.*` restent du **texte pur** (noms en « guillemets », pas de `<strong>`/`<a>` inline) ; toute mise en forme (gras, lien, bouton) vit dans le template JS de `emails.ts`, autour des placeholders.

## Checklist pour ajouter/migrer une chaîne

1. Choisir le namespace de la page.
2. Ajouter la clé dans `fr.json` **et** `en.json`.
3. Remplacer le littéral par `t('…')` (ou `await getTranslations` si async).
4. `npm run build` (le typage des clés est vérifié au build, pas par `tsc --noEmit` seul — voir `.claude/rules/server-architecture.md` pour le piège Turbopack équivalent côté server actions).

Pas de garde-fou automatique (un lint « chaînes en dur » donnerait des milliers de faux positifs ailleurs dans l'app) — c'est une discipline manuelle.
